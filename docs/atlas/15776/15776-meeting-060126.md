# Bug Fix: Decoupling client deliverable update authorization

**Problem**
Authorized users (specifically Neptune facilities and reception roles) are blocked from updating or renaming client deliverable transcript files, resulting in a `403 Forbidden` error. 

This error occurs because the backend's shared `update proceeding file` authorization guard (`update proceeding file off guard` in the Callisto codebase) assumes all file updates are for submission files. It fails to evaluate the request's `is_deliverable` flag and only checks for submission proceeding file permissions. Consequently, users who possess the correct update permissions for client deliverables, but lack equivalent permissions for submission files, are incorrectly blocked by a false-negative authorization check. 

**Requirement**
The system must provide an independent authorization and update path for client deliverable files that is entirely decoupled from submission file permission logic. It must successfully validate update operations for authorized roles (e.g., Neptune facilities) on client deliverables without requiring submission-file-level permissions, while maintaining the structural separation of the client deliverable and submission domains.

**Solution**
Implement the "heavy" architectural split to cleanly separate the client deliverable and submission domains for file updates, mirroring the pattern established by the existing `upload complete` service split. This entails creating a dedicated `UpdateDeliverableFile` authorization guard, a new `UpdateDeliverableFileAction`, and an independent `DeliverableFileService` within the `granting client access` domain. Rather than patching the shared guard, this split ensures that client deliverable renaming is fully isolated, resolving the immediate authorization bug and cleanly supporting future domain-specific capabilities (such as setting sub-track types) without cross-domain regressions.

### Implementation Notes

* **Strategy**: Mirror the domain-separated "heavy split" pattern implemented for the `upload complete` flows (separating submission files from client deliverables).
* **Constraints / Non-goals**: Do not attempt a "light" patch, such as adding conditional `is_deliverable` payload checks to the existing shared `UpdateProceedingFile` guard, as this would violate domain boundaries.
* **Technical Notes**: Define the new `UpdateDeliverableFile` authorization guard within the `granting client access` domain and point the client deliverable rename route to the new `UpdateDeliverableFileAction`.
* **Technical Notes**: Relocate or extract any shared DTOs or helper utilities from the `proceedings` domain to a common/shared layer to prevent circular dependencies.
* **Testing Considerations**: Verify that Neptune facilities can successfully rename client deliverables while still being blocked from renaming submission files.
* **Testing Considerations**: Ensure existing submission file renaming workflows remain completely unaffected by these changes.
* **Estimate**: 5 story points.

### Risks

* **Domain Logic Drift**: Because the renaming and update logic will be split into `DeliverableFileService` and `SubmissionFileService`, any future modifications to core file metadata handling, S3 pathing, or shared properties must be synchronized to prevent functional drift.
* **DTO Coupling**: Moving or surfacing DTOs to support both actions risks introducing circular dependencies between the `proceedings` and `granting client access` domains if the boundaries are not clean.
